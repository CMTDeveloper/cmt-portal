## 1. Goal

Make the v3 visual prototypes functional. After this slice, a real family can:

1. Sign in via emailed magic link (no password).
2. Register a new family — or, if email/phone matches an existing one, be offered to join it instead.
3. Add / edit / remove household members.
4. Invite a co-manager to their family.
5. See their family roster, members, and (still-stub) program/donation state.

The visual layer is done. This slice is the data + auth + API layer underneath, wired to the prototype screens.

## 2. Out of scope

Explicitly NOT in this slice (each becomes its own slice later):
- Bala Vihar enrollment workflow logic (donation-period auto-enrollment, suggested-donation pricing tied to enrollment date).
- Real donation checkout (Stripe / e-transfer flow).
- Teacher / Welcome-team / Admin screens.
- Admin CMS (announcements, donation periods, email templates) — separate slice.
- Migration of the existing `/check-in/*` tree into the new portal.
- Attendance + roster integration with the legacy RTDB.

## 3. Audience

Family role only at this slice. Two sub-roles within a family:
- **Manager** — can edit family, invite, add/remove members. At least one per family.
- **Member** — read-only on the family record + their own profile.

Teacher / Welcome / Admin keep using the existing `/login/*` + `/check-in/*` system unchanged.

## 4. Data model — Firestore

All under the portal Firebase project (`PORTAL_FIREBASE_PROJECT_ID`). Two top-level collections + one lookup index.

### 4.1 `families/{fid}`

```ts
type FamilyDoc = {
  fid: string;                       // generated by the portal at creation
  legacyFid: string | null;          // existing Bala Vihar FID if migrated
  name: string;                      // "Patel"
  location: 'Brampton' | 'Mississauga' | 'Scarborough' | 'Markham';
  createdAt: Timestamp;
  managers: string[];                // Member IDs of managers (denormalized for fast checks)
  searchKeys: string[];              // [lowercase name, fid, legacyFid] — for welcome-team lookup
};
```

### 4.2 `families/{fid}/members/{mid}`

```ts
type MemberDoc = {
  mid: string;                       // `{fid}-{seq}` e.g. "4421-03"
  uid: string | null;                // Firebase Auth UID, set after first sign-in
  firstName: string;
  lastName: string;
  type: 'Adult' | 'Child';
  gender: 'Male' | 'Female' | 'PreferNotToSay';
  manager: boolean;
  joinedAt: Timestamp;
  email: string | null;              // required for adults, optional for children
  phone: string | null;              // required for adults, optional for children
  // Child-only
  schoolGrade: string | null;
  birthMonthYear: string | null;     // "Mar 2017"
  // Adult-only
  volunteeringSkills: string[];
  // Common
  foodAllergies: string | null;
  emergencyContacts: [
    { relation: string; phone: string; email: string },
    { relation: string; phone: string; email: string } | null
  ];
};
```

### 4.3 `contactKeys/{key}`

Lookup index for dedupe + magic-link sign-in. The `key` is `sha256(normalizedContact)` where normalization is:
- email → lowercase + trim
- phone → digits only, prefixed with `+1` for Canadian numbers

```ts
type ContactKeyDoc = {
  contactKey: string;                // the sha256 hash (also the doc id)
  type: 'email' | 'phone';
  fid: string;                       // which family this contact belongs to
  mid: string;                       // which member specifically
};
```

Both an email and a phone produce separate `contactKey` docs pointing at the same `(fid, mid)` pair. A registered family with one adult having email + phone creates **two** contact-key docs.

### 4.4 Why contact-key as its own collection (not embedded)

- **O(1) dedupe lookup.** During registration: hash the entered email, do `getDoc(doc('contactKeys', hash))`. That's it. No range scans, no indexing the entire families collection by email.
- **Atomic invariants.** Family registration writes contactKey docs inside the same transaction as the family + member creation — guarantees no duplicates can sneak in under concurrent registrations.
- **Privacy preserves.** Raw email/phone never indexed; just the hash. The hash is unsalted (deliberate — same contact must produce the same hash across redeploys) but the contactKey alone is not a credential and the doc only contains `(fid, mid)` references, no PII.

## 5. Auth flow — 6-digit OTP via existing CMT infra

**Decision (resolved):** No Firebase magic-link. We reuse the existing OTP code pattern already running in the `/check-in/*` flow — same primitives, same SES templates, same SNS topics. The pipeline (`normalizeContact`, `storeVerificationCode`, `checkAndRecordOtpRateLimit`, `resolveSender`) is in `apps/portal/src/features/check-in/shared/` and the routes `/api/auth/family/send-code` + `/api/auth/family/verify-code` are the working reference implementation.

### 5.1 Sign-in (existing user) — email OR phone

```
USER                  PORTAL                            AWS SES / SNS
 │                      │                                    │
 │ POST /api/setu/auth/send-code { type, value }            │
 │ ───────────────────► │                                    │
 │                      │ normalizeContact(type, value)      │
 │                      │ checkAndRecordOtpRateLimit         │
 │                      │ findSetuFamilyByContact (new)     │
 │                      │   ├ contactKeys/{hash} lookup     │
 │                      │   └ if no Setu hit, fall back     │
 │                      │     to legacy RTDB roster         │
 │                      │     (so pre-migration families    │
 │                      │      can still sign in)           │
 │                      │ generateCode (6-digit)            │
 │                      │ storeVerificationCode(hash, code) │
 │                      │ resolveSender().sendEmail/SMS     │
 │                      │ ─────────────────────────────────► │
 │ ◄─── 200 OK          │                                    │
 │                      │                                    │
 │  (user enters code on /sign-in)                          │
 │                                                            │
 │ POST /api/setu/auth/verify-code { type, value, code }    │
 │ ───────────────────► │                                    │
 │                      │ consumeVerificationCode            │
 │                      │ → if Setu hit: load family, mid    │
 │                      │ → if legacy hit but no Setu doc:   │
 │                      │     auto-migrate this family now   │
 │                      │     (single-family migration)      │
 │                      │ → if no hit at all:                │
 │                      │     create stub session, 302 to    │
 │                      │     /register?contact=verified     │
 │                      │ createPortalSessionCookie          │
 │                      │ Set-Cookie: __session=…            │
 │                      │ 302 → /family or /register         │
```

Note the **legacy fallback in send-code**: a family that was migrated overnight gets a Setu contactKey hit. A family NOT yet migrated falls back to the legacy roster and STILL receives a code — they don't get blocked. The verify step then auto-migrates them on first successful sign-in (one family at a time, lazily). This makes the legacy → Setu migration low-stakes; a bulk migration script can still pre-populate (Section 11), but lazy single-family migration covers the long tail.

### 5.2 Session shape

Reuse `@cmt/firebase-shared/admin/session.createPortalSessionCookie`. Extend `SessionClaims` in `@cmt/shared-domain/auth`:

```ts
type SessionClaims = {
  uid: string;
  role:
    | 'family'       // legacy check-in family (existing)
    | 'teacher'      // existing
    | 'admin'        // existing
    | 'family-manager'  // NEW — Setu manager
    | 'family-member'   // NEW — Setu non-manager
    | 'welcome-team';   // NEW — welcome-team / admin sub-role
  fid?: string;      // Setu family id (was `familyId`)
  mid?: string;      // Setu member id — NEW
  familyId?: string; // legacy alias, kept for /check-in/* compat
};
```

Both old and new claim shapes coexist. Legacy `family` role keeps `familyId`; new `family-manager` / `family-member` use `fid` + `mid`.

### 5.3 Sign-out

Add `POST /api/setu/auth/signout` that clears the cookie and 303s to `/` (Setu landing). Existing `/api/auth/signout` continues to 303 to `/login` for the legacy flow. Two separate endpoints since the destinations differ.

### 5.4 Phone-OTP fallback

The same `/api/setu/auth/send-code` accepts `{ type: 'phone', value }` and routes via `resolveSender().sendSMS` (AWS SNS) — exactly the existing check-in pattern. The `/sign-in` UI shows both an email field and a "Use phone number instead" toggle (already in the v3 design mockup). No separate endpoint, no separate session shape — just a different `type` arg on the same flow.

## 6. Registration & dedupe flow

### 6.1 Step 1 — contact lookup

`POST /api/setu/family-lookup`

Request:
```ts
{ email: string; phone: string }
```

Response (200):
```ts
| { match: null }
| { match: { fid: string; name: string; location: string; memberCount: number; managerInitials: string } }
```

Server:
1. Validate email/phone with zod.
2. Compute both contactKey hashes.
3. Read both `contactKeys/{hash}` docs in parallel.
4. If either resolves to a family, fetch that family + a redacted summary (name, location, member count, manager initials).
5. If neither matches, return `{ match: null }`.

Rate-limited per IP (e.g. 10/min) to prevent contact enumeration.

### 6.2 Step 2a — new family registration

`POST /api/setu/register`

Request:
```ts
{
  email: string;
  phone: string;
  familyName: string;
  location: 'Brampton' | 'Mississauga' | 'Scarborough' | 'Markham';
  manager: { firstName: string; lastName: string; gender: 'Male'|'Female'|'PreferNotToSay'; };
  additionalMembers: Array<{
    firstName: string;
    lastName: string;
    type: 'Adult' | 'Child';
    gender: 'Male'|'Female'|'PreferNotToSay';
    schoolGrade?: string;
    birthMonthYear?: string;
    foodAllergies?: string;
    email?: string;
    phone?: string;
  }>;
};
```

Server (all in one Firestore transaction):
1. Hash both contact strings; verify neither contactKey doc exists (race-safe duplicate prevention).
2. Generate `fid` (12-char base32, e.g. `4421ABCDEF`).
3. Create `families/{fid}` doc.
4. Create `families/{fid}/members/{fid}-01` (the manager).
5. For each additional member, create `families/{fid}/members/{fid}-NN`.
6. For each member with an email or phone, create `contactKeys/{hash}` doc.
7. Return `{ fid, mid }`.

If the user wasn't signed in via magic link before reaching step 2 (typical flow: they arrived at /register without auth), the response also creates a session cookie so they land at `/family` authenticated.

### 6.3 Step 2b — join existing family

`POST /api/setu/family/join`

Request: `{ fid: string; contactProof: { type: 'email'|'phone'; value: string } }`

Server:
1. Hash `contactProof.value`; verify `contactKeys/{hash}.fid === fid`.
2. If the user isn't already a member of that family, create a new member with `manager: false` and link the contactKey to the new mid.
3. Return session cookie + redirect to `/family`.

In the prototype the "Join the Patel family →" button just navigates to /family — Slice 2 wires it to call this endpoint.

## 7. Invite flow

A manager invites another adult (typically the spouse) to co-manage the family.

`POST /api/setu/invite/send` (auth required, manager role)

Request: `{ email: string; relation: string; }`

Server:
1. Generate single-use invite token (`crypto.randomBytes(24).toString('base64url')`).
2. Write `families/{fid}/invites/{token}` with `{ email, relation, createdAt, expiresAt: now + 14d, acceptedAt: null }`.
3. Send invite email (reuse AWS SES helpers in `apps/portal/src/lib/aws/ses.ts`) with link to `/invite/{token}`.

**Invite TTL — 14 days (decision, with rationale).** Industry norms for non-security-critical "team/household invite" links cluster around 7–30 days:

| Service | TTL |
|---|---|
| GitHub org invite | 7 days |
| Google Workspace user invite | 7 days |
| Slack workspace invite | 30 days (link); never (single-use email invite) |
| Microsoft Teams team invite | 30 days |
| Linear workspace invite | 7 days |
| Notion workspace invite | never (link); 30 days (single-use email invite) |
| AWS IAM password-set link | 7 days |
| Stripe account invite | 24 hours (security-tightened) |

7 days is the safe default; 30 days is on the generous end. For Setu — volunteer-managed, low-stakes (a wrongly-accepted invite is undoable by the manager), and families often handle non-urgent things on weekends — **14 days** is the sweet spot. Long enough to survive a busy week + travel; short enough that stale invites don't accumulate. Configurable via env (`SETU_INVITE_TTL_DAYS`, default 14).

`POST /api/setu/invite/accept` (auth required — must be signed in via OTP code first)

Request: `{ token: string }`

Server:
1. Read `families/{fid}/invites/{token}`.
2. Verify it exists, not expired, not already accepted.
3. Verify the signed-in user's verified email matches the invite's `email`.
4. Add the user as a new member (manager: true).
5. Mark invite accepted.

## 9. Middleware changes

`apps/portal/src/middleware.ts` updates:

1. Move `/family`, `/family/`, `/sign-in/verify` OUT of `PUBLIC_ROUTES`. Keep `/sign-in`, `/register`, `/register/family`, `/invite/:token` public (they're the entry points). Keep `/api/setu/auth/send-link`, `/api/setu/family-lookup`, `/api/setu/auth/verify` public.
2. Extend `canAccessRoute` to understand `family-manager` and `family-member` roles. Both can access `/family/*`. Only `family-manager` can hit the `POST/PATCH/DELETE` endpoints under `/api/setu/members*`.
3. The middleware reads the session cookie; if missing or invalid AND the route is `/family/*`, redirect to `/sign-in` (not `/login`!) with `?from=…&error=session-expired`.
4. The legacy `/login` redirect for `/check-in/*` paths stays unchanged.

## 11. Legacy FID migration

The legacy Bala Vihar roster lives in the master Firebase RTDB (the `chinmaya-family-check-in` app's data source). Its schema is row-keyed students with `fname`/`lname`/`pemail`/`phphone`/`payment`/`grade` fields, where `grade === 99` marks a parent row.

### Strategy

One-time, idempotent migration script run from `apps/portal/scripts/migrate-legacy-families.ts`:

1. Read the entire legacy roster from RTDB (existing `apps/portal/src/features/check-in/shared/rtdb/family-lookup.ts` parser can be reused — it already adapts the legacy shape into a `Family` type).
2. For each legacy family:
   - Skip if a `families/*` doc already exists with `legacyFid === oldFid`.
   - Otherwise create a fresh Setu family with the same household: new `fid` (generated), `legacyFid` = old FID, members ported with best-effort field mapping.
   - Write contactKey docs for any emails/phones.
3. Idempotent: re-running is safe; already-migrated families are skipped.
4. Output: a CSV report (`migrated.csv`) of `oldFid → newFid` mappings for sevak reference.

### When to run

- Twice locally against UAT (`chinmaya-setu-uat`) to verify.
- Once against production *immediately before* enabling the new Setu sign-in for real families.
- Old `/check-in/*` keeps working off the legacy RTDB during the transition — the migration is read-only from the legacy side.

### Search by legacy FID (welcome-team)

The roster screen and welcome-team search (designed in the brief, not built yet in code) must look up by old FID. Strategy: the family search query (`/api/setu/family/search?q=…`) checks both `fid ==` and `legacyFid ==` and merges. Both fields are indexed.

## 14. Risks & tradeoffs

| Risk | Mitigation |
|---|---|
| **Hash collisions on contactKey** | SHA-256 → collision space is 2^256. Not a real risk; document it anyway. |
| **A malicious user could enumerate families** by guessing emails | Rate-limit `/family-lookup` per IP + per contact-hash; return generic responses for missing matches (no timing-attack delta). |
| **Magic-link emails go to spam** | Use the existing AWS SES setup with the production CMT sender domain. Configure SPF/DKIM in env. |
| **A user has multiple legacy FIDs** for the same household (RTDB data error) | Migration script flags these as conflicts; sevak resolves manually before re-running. |
| **A migrated family signs in via magic link to a different email than the one in the legacy roster** | They land at `/register` (dedupe misses), would create a duplicate. Mitigation: family-lookup should also check `legacyFid` if user provides it; OR add a "I already had a FID — link my account" affordance on /register. (Worth designing in 2b.) |
| **Concurrent registration race** | Single Firestore transaction guarantees atomicity. Re-verify contactKey absence inside the txn. |

## 17. Release strategy

Slice 2 lands incrementally to `main` (sub-slices 2a → 2f), but **is NOT announced to real families until the full Bala Vihar registration flow is end-to-end functional** — that means Slices 3 and 4 also complete:

| Slice | What it ships | Required for launch? |
|---|---|---|
| **Slice 2** (this one) | Setu auth + family registration + dedupe + member CRUD + invite + welcome-team search + legacy migration | Yes |
| **Slice 3** | Bala Vihar enrollment workflow (auto-enroll on first attendance, donation-period pricing snapshot) + Stripe / e-Transfer donation checkout + tax-receipt PDF generation | Yes |
| **Slice 4** | Teacher views (class roster, take attendance, per-student stats, add-on-prompt invite) + welcome-team full screens + admin (donation periods, announcements, email templates) | Yes — at least teacher core flows |
| **Slice 5** | Cutover from `/check-in/*` to Setu (flip `NEXT_PUBLIC_FEATURE_CHECK_IN_KIOSK`, deprecate `/login/*`) | No — can ship after |

Once Slices 2 + 3 + 4 are merged and soaked in UAT, a single "announce to families" moment with the family invite email + the legacy → Setu link sent to all known contacts.

Until that announcement, the new Setu routes are publicly reachable (since they're in `PUBLIC_ROUTES` for the visual prototype phase) — but no families know about them. The legacy `/login` + `/check-in/*` path remains the production entry point for sevaks and BV families.
